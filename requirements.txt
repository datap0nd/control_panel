IMPORTANT STATUS AND FILENAME RULES

Status is the first filename attribute.

The only accepted Status values are:

- capture
- capturefailed

The filename format is:

Status__Sub__App__KPI__YYYYMMDD_HHMMSS.png

Examples:

capture__Sub-1__App-1__KPI-A__20260720_165748.png
capture__Sub-1__App-1__KPI-B__20260720_165752.png
capturefailed__Sub-1__App-1__KPI-C__20260720_165801.png

The filename is the screenshot identifier. A separate Capture ID is not required.

There will be approximately five KPIs for every Sub and App combination. This produces approximately five rows for each Sub and App test run.


PROMPT CSV RULES

The prompt CSV columns are:

sub
app
kpi
prompt_version
prompt

Example:

sub,app,kpi,prompt_version,prompt
Sub-1,App-1,KPI-A,1,"Check whether the required visual elements are displayed correctly."

Every Sub, App, and KPI combination must have one initial prompt row.

All initial prompts will use prompt_version 1.

When a prompt changes later:

- Add another row for the same Sub, App, and KPI combination.
- Increase prompt_version from 1 to 2, then 3, and so on.
- Do not delete or overwrite previous prompt versions.
- Use the highest prompt_version by default for new screenshots.
- Do not automatically reprocess historical screenshots.
- Keep the prompt_version on every SQL result so it is clear which prompt generated that result.

Status is not part of prompt selection.

Files with Status capture use Sub, App, and KPI to select the latest prompt.

Files with Status capturefailed skip prompt selection and visual evaluation.

EMAIL TO THE DATABASE CONSUMER

Subject: Proposed screenshot evaluation table

Hi,

I am defining the output table for the screenshot-validation process.

The table will be:

REPORTING_SCHEMA.screenshot_evaluations

The process will update the table daily.

I propose keeping the full evaluation history. Existing results will not be overwritten.

When a prompt changes, the latest prompt version will be used for new screenshots. Historical screenshots will not be automatically processed again. Each result will record the prompt version that generated it.

Each row will represent one screenshot evaluation for a specific Sub, App, and KPI.

The proposed columns are:

- evaluation_id: Unique identifier for the evaluation
- evaluation_key: Prevents the same evaluation from being created twice
- screenshot_id: Complete source filename without the .png extension
- source_filename: Complete source filename including the .png extension
- image_sha256: Screenshot fingerprint used for audit and integrity checks
- capture_status: capture or capturefailed, taken from the first filename attribute
- sub: Sub from the screenshot filename
- app: App from the screenshot filename
- kpi: KPI from the screenshot filename
- captured_at: Date and time when the screenshot was captured
- processing_status: Indicates whether technical processing succeeded
- validation_result: Pass, fail, or blank when no valid visual result exists
- validation_reason: Reason provided for the pass or fail result
- prompt_version: Version of the prompt used for the evaluation
- model_id: Model version used for the evaluation
- attempt_count: Number of processing attempts
- error_code: Technical error category
- error_message: Technical error details
- created_at: Date and time when the evaluation row was created
- completed_at: Date and time when processing finished

Technical errors will be stored in the same table. They will be recorded as follows:

processing_status = error
validation_result = blank
error_code = populated
error_message = populated

This ensures that a technical processing problem is not incorrectly reported as a visual failure.

If screenshot capture fails before visual validation, the row will be recorded as follows:

capture_status = capturefailed
processing_status = skipped
validation_result = blank

Raw model responses will not be stored in the reporting table.

Please confirm whether you require any additional columns beyond those listed above.

Thanks.
