Subject: New automated report alert capability now available

Hi team,

The new Recurrences feature is now live in [INTERNAL TOOL] under Tools > Recurrences.

This gives us a controlled way to create scheduled alerts from table or matrix visuals in [BI PLATFORM] and notify users when the returned data meets defined conditions. The main use case is replacing repetitive manual report checks with scheduled, traceable notifications.

What the feature does

- Connects an alert to a specific report, page, and table or matrix visual.
- Reads the current summarized data behind that saved visual.
- Applies optional row-level rules, such as sending only rows where a value exceeds a threshold.
- Can send all matching rows to one recipient list, or split the results into subgroups and send a separate email for each subgroup.
- Formats the returned rows as an HTML table and includes a direct link to the report.
- Supports daily, weekday, weekly, and monthly schedules.
- Records each run, including its status, row counts, email count, and failure reason.
- Sends nothing when there are no matching rows.

Authentication

The feature reuses the work account already connected through the [INTERNAL TOOL] scanner. It does not require us to paste access tokens into scripts or configure a separate login for every alert.

The initial account connection creates a delegated authentication cache on the machine running [INTERNAL TOOL]. The application requests an access token from that cache and silently renews it when Microsoft authentication permits. The token remains inside the application process and is never included in the alert configuration or email.

This means the connected account must already have access to the relevant workspace, report, and semantic model. If the cached session expires or requires interactive authentication, the scanner will show that the account must be reconnected.

How the [BI PLATFORM] APIs are used

The implementation uses supported [BI PLATFORM] interfaces rather than automating clicks in the report:

- The REST API lists the reports available in the configured workspace.
- The REST API retrieves the stable technical page identifiers for the selected report.
- Before every alert run, the REST API checks the semantic model's latest refresh attempt. The alert proceeds only when the latest status is Completed.
- The official embedded client identifies the exact saved visual and first requests its summarized export data.
- If the visual export API cannot run the visual's data query, [INTERNAL TOOL] reads the selected visual's current field bindings and filters, generates the equivalent query, and uses the semantic-model Execute Queries REST endpoint as a fallback.

The fallback requires the connected account to have Read and Build access to the semantic model. The tenant setting that permits the Execute Queries REST API must also be enabled. If those permissions are unavailable, the alert fails closed and sends no user data.

How to create an alert

1. Open [INTERNAL TOOL] and go to Tools > Recurrences.
2. Select Create new.
3. Choose the report from the configured workspace.
4. Select the report page.
5. Load the available visuals and choose the required table or matrix.
6. Fetch the preview and confirm that the expected columns and rows are returned.
7. Choose the delivery method:
   - Single email sends all eligible rows to one recipient list.
   - Subgroups split the rows using a selected column. Each subgroup can have its own display name and recipients.
8. Add row rules if required. Rules are optional. Multiple rules are evaluated together, and only matching rows are included.
9. Confirm the alert owner. It defaults to the owner of the selected report.
10. Confirm that the owner has a valid email in Management > Create > People.
11. Set the email subject and schedule.
12. Save the recurrence.
13. Use Create drafts to test the complete refresh, extraction, filtering, and formatting path without sending the alert.
14. After reviewing the draft, use Run now for a controlled live test.

Ownership and failures

The alert owner defaults to the report owner, although another registered owner can be selected. Owner email addresses are resolved from the central People registry, so email corrections apply without editing every alert.

If an actual send run fails before delivery, the owner receives a separate failure email containing:

- The alert and report name.
- The selected page and visual.
- The latest refresh status when available.
- The recorded failure reason.
- A direct link to the report.

The alert will not send user data when the latest refresh failed, is still running, was cancelled, has no history, or cannot be verified. It also fails closed if a required rule or subgroup column disappears, the visual is deleted, permissions change, or the result cannot be reproduced safely.

Operational notes

- The machine hosting [INTERNAL TOOL] must be running.
- The connected work account must retain access to the report and semantic model.
- The desktop email profile used by the existing alert process must remain configured.
- Adding or removing ordinary columns from the saved table should be picked up automatically because each run reads the current visual structure.
- If a column used by a rule or subgroup is removed, the run fails instead of sending an incorrect result.
- Failure notifications use the same desktop email delivery mechanism. If that email client is completely unavailable, the failure and notification attempt are still recorded, but the system cannot email its own outage.

This is now available for us to test with suitable operational reports. Please use Create drafts first and validate the recipients, filtering, table layout, and report link before enabling scheduled delivery.
