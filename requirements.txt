SCREENSHOT VALIDATION - AGREED FLOW


1. PURPOSE

The process will run daily, read screenshots from a permanent shared folder, select the correct visual criteria, evaluate each successfully captured screenshot with the configured vision model, and write the result to SQL.

The process finishes only when the SQL result has been written successfully.


2. RESPONSIBILITIES

Business criteria coordinator

- Contacts the relevant business teams.
- Obtains the criteria for each Region, Operator, and Stage combination.
- Business criteria collection is already in progress.

Prompt owner

- Receives the business criteria.
- Creates the prompt criteria files.
- Completes the prompt mapping.
- Owns prompt versions and prompt changes.
- Coordinates any labelled pass and fail examples needed for prompt testing.

Screenshot process owners

- Produce the screenshots.
- Save the images as PNG files.
- Apply the agreed filename convention.
- Provide the permanent shared-folder path.
- Explain what artifact is produced when capture fails.
- Provide a few unlabelled sample PNGs for technical ingestion testing.

Pipeline owner

- Creates the mapping CSV template.
- Reads and validates the screenshot filenames.
- Selects the correct prompt.
- Calls the configured vision model.
- Validates the model response.
- Writes the final evaluation to SQL.
- Prevents duplicate processing.

Database consumer

- Consumes the daily output table.
- Confirms whether any additional reporting columns are required.


3. SCREENSHOT IDENTIFIER AND FILENAME

The filename is the screenshot identifier. A separate Capture ID is not required.

The attributes must appear in this order:

1. Status
2. Region
3. Operator
4. Stage
5. Capture date and time

Filename format:

Status__Region__Operator__Stage__YYYYMMDD_HHMMSS.png

The only accepted Status values are:

- capture
- capturefailed

Examples:

capture__Region-1__Operator-1__Stage-A__20260720_165748.png
capture__Region-1__Operator-1__Stage-B__20260720_165752.png
capturefailed__Region-1__Operator-1__Stage-C__20260720_165801.png

The screenshot_id stored in SQL will be the complete filename without the .png extension.

There will be approximately five Stages for every Region and Operator combination. This produces approximately five evaluation rows for each Region and Operator test run.

The Region, Operator, and Stage values in the filename must match the values in the prompt mapping.

The design assumes that the screenshot process will not create two files with the same Status, Region, Operator, Stage, and timestamp within the same second.


4. SHARED-FOLDER INPUT

The screenshot process owners will provide the path to one permanent shared project folder.

The daily pipeline will read new PNG files from that folder.

The screenshot process owners do not need to provide labelled good or bad examples. The pipeline owner only needs two or three representative PNGs to test that the files can be found, opened, parsed, and sent correctly.

One point must still be confirmed: when Status is capturefailed, the screenshot process must produce an artifact that the daily pipeline can discover. If no file or record is produced, the pipeline has no way to detect or store the capture failure.


5. PROMPT MAPPING

The pipeline owner will create the mapping CSV template.

The mapping columns are:

region
operator
stage
prompt_file
prompt_version
enabled

Example row:

Region-1,Operator-1,Stage-A,prompts/region-1_operator-1_stage-a.txt,1,true

Status is not part of the prompt mapping.

Files with Status capture use the Region, Operator, and Stage values to select a prompt.

Files with Status capturefailed skip prompt selection and skip the vision model.

The prompt owner may reuse the same prompt file for multiple combinations when the criteria are identical.

When prompt criteria change, the prompt owner must increase prompt_version. Previous evaluation rows remain in SQL as history.

A missing, disabled, or duplicated mapping is a technical processing error. It is not a visual failure.


6. MODEL PROCESSING

For a file with Status capture, the pipeline will:

1. Validate that the file is a readable PNG.
2. Parse Status, Region, Operator, Stage, and captured time from the filename.
3. Find exactly one enabled mapping for the Region, Operator, and Stage.
4. Load the mapped prompt criteria file.
5. Add the fixed technical instructions and response format.
6. Send the screenshot and criteria to the configured vision model.
7. Accept only a structured pass or fail result with a non-empty reason.
8. Write the final evaluation to SQL.

The valid visual results are:

- pass
- fail

A timeout, unavailable service, invalid response, missing mapping, or unreadable PNG is a technical processing error. A technical processing error must never be converted into a visual fail.


7. CAPTURE-FAILURE PROCESSING

For an input with Status capturefailed, the pipeline will:

1. Parse the available filename attributes.
2. Skip prompt selection.
3. Skip the vision model.
4. Write a SQL row showing that capture failed and validation was not performed.

The output will be:

capture_status = capturefailed
processing_status = skipped
validation_result = blank


8. SQL OUTPUT

Target table:

REPORTING_SCHEMA.screenshot_evaluations

Update frequency:

Daily

History:

Full history will be retained. Existing evaluations will not be overwritten. Re-evaluation with a different prompt version or model version creates a new evaluation row.

Row definition:

One row represents one screenshot evaluation using one prompt version and one model version.

Proposed columns:

- evaluation_id: Unique identifier for the evaluation
- evaluation_key: Prevents the same evaluation from being created twice
- screenshot_id: Complete source filename without the .png extension
- source_filename: Complete source filename including the .png extension
- image_sha256: Screenshot fingerprint for integrity and audit
- capture_status: capture or capturefailed, taken from the first filename attribute
- region: Region taken from the filename
- operator: Operator taken from the filename
- stage: Stage taken from the filename
- captured_at: Capture date and time taken from the filename
- processing_status: succeeded, skipped, or error
- validation_result: pass, fail, or blank
- validation_reason: Reason for a valid pass or fail result
- prompt_version: Prompt version used for the evaluation
- model_id: Model version used for the evaluation
- attempt_count: Number of processing attempts
- error_code: Technical error category
- error_message: Technical error detail
- created_at: Date and time when the evaluation row was created
- completed_at: Date and time when processing finished

Raw model responses will not be stored in the reporting table.

Technical errors will be stored as:

processing_status = error
validation_result = blank
error_code = populated
error_message = populated


9. DAILY END-TO-END FLOW

1. The screenshot process saves the daily files in the permanent shared folder.

2. The pipeline discovers PNG files that have not already been processed for the same prompt and model version.

3. The pipeline validates and parses the filename.

4. If Status is capturefailed, the pipeline skips visual validation and writes a skipped row to SQL.

5. If Status is capture, the pipeline finds the prompt using Region, Operator, and Stage.

6. The pipeline sends the screenshot and selected criteria to the configured vision model.

7. A valid pass or fail result is written to SQL with the reason.

8. A technical error is written to SQL with a blank validation result and populated error details.

9. The database consumer receives the updated full-history table on the daily schedule.

Subject: Proposed screenshot evaluation table

Hi,

I am defining the output table for the screenshot-validation process.

The proposed table is:

REPORTING_SCHEMA.screenshot_evaluations

The table will be updated daily.

I propose keeping the full evaluation history. Existing results will not be overwritten. If a screenshot is evaluated again with a different prompt version or model version, it will create a new evaluation row.

Each row will represent one screenshot evaluation for a specific Region, Operator, and Stage.

The proposed columns are:

- evaluation_id
- evaluation_key
- screenshot_id
- source_filename
- image_sha256
- capture_status
- region
- operator
- stage
- captured_at
- processing_status
- validation_result
- validation_reason
- prompt_version
- model_id
- attempt_count
- error_code
- error_message
- created_at
- completed_at

The capture_status field will contain capture or capturefailed, based on the first attribute of the screenshot filename.

Technical errors will be stored with processing_status set to error, validation_result left blank, and error_code and error_message populated. This prevents a technical problem from being reported as a visual failure.

Capture failures will be stored with capture_status set to capturefailed, processing_status set to skipped, and validation_result left blank.

Raw model responses will not be stored in the reporting table.

Please confirm whether you require any additional columns beyond those listed above.
